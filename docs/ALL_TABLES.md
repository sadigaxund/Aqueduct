# Aqueduct Observability — All Tables Reference

Aqueduct writes runtime state into three logical stores: **observability**,
**lineage**, and **depot**. Phase 28 introduced pluggable backends for each.
Default DuckDB layout (per-blueprint):

```
.aqueduct/
  obs.db          ← run records, failures, healing, signals, probes, metrics
  lineage.db      ← column-level lineage
  depot.db        ← cross-run KV state (@aq.depot.*)
  snapshots/      ← schema_snapshot JSON files (one per probe per run)
  watermarks/     ← {blueprint_id}__{channel_id}.json sidecars
  checkpoints/    ← Parquet checkpoint dirs per --resume run_id
```

When `backend: postgres` is configured for a store, the tables documented
below live in a per-store schema (`obs.*`, `lineage.*`, `depot.*`) inside
a single Postgres database. When `backend: redis` is configured for the
depot, the `depot_kv` keys live directly in the configured Redis database
without a relational table. SQL examples below run against DuckDB and
Postgres unchanged — the engine uses standard ANSI SQL and rewrites
`?` placeholders to `%s` for Postgres internally.

Every CREATE TABLE statement listed below uses `IF NOT EXISTS`, so attaching to
an existing store is non-destructive. Aqueduct also performs additive
schema migrations on startup (see `aqueduct/surveyor/surveyor.py:_DDL`).

## `obs.db` — Observability store

Single DuckDB file per blueprint (Phase 20: `.aqueduct/obs/{blueprint_id}/obs.db`).
Opened by both the Surveyor (write path) and the CLI / `aqueduct doctor` (read path).

### `run_records`

**Source:** `aqueduct/surveyor/surveyor.py:41` — written on `Surveyor.start()` and updated on `Surveyor.record()`.

| Column | Type | Notes |
|---|---|---|
| `run_id` | VARCHAR PRIMARY KEY | UUID — generated by CLI or supplied via `--run-id` |
| `blueprint_id` | VARCHAR NOT NULL | From `Blueprint.id` |
| `status` | VARCHAR NOT NULL | `running` → `success` / `error` / `patched` / `skipped` |
| `started_at` | TIMESTAMPTZ NOT NULL | UTC, ISO-8601 |
| `finished_at` | TIMESTAMPTZ | UTC; null while running |
| `module_results` | JSON | List of `ModuleResult` snapshots |

```sql
-- Recent failures across all runs
SELECT run_id, started_at, finished_at
FROM run_records
WHERE status = 'error'
ORDER BY started_at DESC LIMIT 10;
```

### `failure_contexts`

**Source:** `aqueduct/surveyor/surveyor.py:50` — one row per failed run, written by `Surveyor.record()` when the run ended with status=error.

| Column | Type | Notes |
|---|---|---|
| `run_id` | VARCHAR PRIMARY KEY | Foreign key to `run_records.run_id` |
| `blueprint_id` | VARCHAR NOT NULL | |
| `failed_module` | VARCHAR NOT NULL | Module ID of the first error |
| `error_message` | VARCHAR NOT NULL | First module's `ModuleResult.error` |
| `stack_trace` | VARCHAR | Truncated to `_STACK_TRACE_MAX_LINES` lines in LLM prompts |
| `manifest_json` | JSON | Full compiled Manifest at the moment of failure |
| `provenance_json` | JSON | `ProvenanceMap.to_dict()` slice (added by migration) |
| `started_at` | TIMESTAMPTZ NOT NULL | |
| `finished_at` | TIMESTAMPTZ NOT NULL | |

```sql
-- Pull the error message + provenance for one run
SELECT failed_module, error_message,
       json_extract(provenance_json, '$.context') AS context_prov
FROM failure_contexts WHERE run_id = ?;
```

### `healing_outcomes`

**Source:** `aqueduct/surveyor/surveyor.py:62` — written by `Surveyor.record_healing_outcome()` at every patch decision point in the self-healing loop (guardrail block, human stage, auto apply, aggressive apply).

| Column | Type | Notes |
|---|---|---|
| `id` | VARCHAR PRIMARY KEY | UUID (was INTEGER pre-migration) |
| `run_id` | VARCHAR NOT NULL | Run that triggered healing |
| `failed_module` | VARCHAR | |
| `failure_category` | VARCHAR | LLM-assigned, e.g. `schema_drift`, `bad_path` |
| `model` | VARCHAR | LLM model name from `agent.model` |
| `patch_id` | VARCHAR | PatchSpec identifier |
| `confidence` | DOUBLE | 0.0–1.0 |
| `patch_applied` | BOOLEAN | False if guardrail blocked or human-staged |
| `run_success_after_patch` | BOOLEAN | True only when re-run reached success |
| `applied_at` | VARCHAR | ISO-8601 UTC (string, not TIMESTAMPTZ) — used by rate-limit query |

```sql
-- LLM accuracy over the last week
SELECT model,
       COUNT(*) AS attempts,
       SUM(CASE WHEN run_success_after_patch THEN 1 ELSE 0 END) AS fixed,
       AVG(confidence) AS avg_conf
FROM healing_outcomes
WHERE applied_at >= (CURRENT_TIMESTAMP - INTERVAL 7 DAY)::VARCHAR
GROUP BY model
ORDER BY attempts DESC;
```

### `patch_simulation`

**Source:** `aqueduct/surveyor/surveyor.py:_DDL` (added Phase 29a, extended Phase 29b). One row per Gate-2, Gate-3, or Gate-4 evaluation — `aqueduct patch preview` and the auto/aggressive self-healing loop both write here.

| Column | Type | Notes |
|---|---|---|
| `id` | VARCHAR PRIMARY KEY | UUID per evaluation |
| `run_id` | VARCHAR | Failing run that triggered healing (null for direct `patch preview`) |
| `blueprint_id` | VARCHAR | |
| `patch_id` | VARCHAR NOT NULL | Matches `patches/applied/{id}.json` |
| `gate` | VARCHAR NOT NULL | `gate2` (lineage), `gate3` (sandbox replay), `gate4` (explain regression) |
| `status` | VARCHAR NOT NULL | `pass` / `fail` / `warn` / `skip` |
| `detail` | VARCHAR | Free-text — failing rule, missing column, executor error, plan regression list |
| `sample_rows` | BIGINT | Per-Ingress LIMIT used by Gate 3 (NULL for Gate 2 / Gate 4) |
| `duration_ms` | BIGINT | Wall-clock for the gate |
| `recorded_at` | VARCHAR | ISO-8601 UTC |

```sql
-- Were recent patches blocked by Gate 3 sandboxes?
SELECT patch_id, status, detail, recorded_at
FROM patch_simulation
WHERE gate = 'gate3' AND status = 'fail'
ORDER BY recorded_at DESC LIMIT 20;
```

### `explain_snapshot`

**Source:** `aqueduct/surveyor/surveyor.py:_EXPLAIN_SNAPSHOT_DDL` (added Phase 29b). Rolling per-module physical-plan baseline. Written by the executor after every successful module run (non-Egress). Read by Gate 4 (`aqueduct/patch/explain_gate.py`) as the pre-patch baseline. Surveyor prunes to the most recent `keep_last_n` (default 5) rows per `(blueprint_id, module_id)`.

| Column | Type | Notes |
|---|---|---|
| `blueprint_id` | VARCHAR NOT NULL | PK with `run_id`, `module_id` |
| `run_id` | VARCHAR NOT NULL | |
| `module_id` | VARCHAR NOT NULL | Non-Egress module whose DataFrame plan was captured |
| `captured_at` | VARCHAR NOT NULL | ISO-8601 UTC |
| `exchange_count` | INTEGER NOT NULL | Count of `Exchange` nodes (shuffle proxy) |
| `python_udf_count` | INTEGER NOT NULL | Count of `BatchEvalPython` nodes |
| `broadcast_count` | INTEGER NOT NULL | Count of `BroadcastExchange` nodes |
| `plan_text` | VARCHAR NOT NULL | Full `df.explain(mode='formatted')` text — kept for debugging |

```sql
-- Latest baseline counts per module
SELECT blueprint_id, module_id,
       exchange_count, python_udf_count, broadcast_count, captured_at
FROM explain_snapshot
WHERE (blueprint_id, module_id, captured_at) IN (
  SELECT blueprint_id, module_id, MAX(captured_at)
  FROM explain_snapshot
  GROUP BY 1, 2
);
```

## `lineage.db` — Column-level lineage

### `column_lineage`

**Source:** `aqueduct/compiler/lineage.py:28` — populated after a successful run by walking SQL Channel queries with `sqlglot.lineage`.

| Column | Type | Notes |
|---|---|---|
| `blueprint_id` | VARCHAR NOT NULL | |
| `run_id` | VARCHAR NOT NULL | |
| `channel_id` | VARCHAR NOT NULL | Module ID of the Channel that produced the output column |
| `output_column` | VARCHAR NOT NULL | |
| `source_table` | VARCHAR NOT NULL | Spark temp view name (= upstream module ID) |
| `source_column` | VARCHAR NOT NULL | |
| `captured_at` | TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP | |

Index: `idx_lineage_channel` on `(blueprint_id, channel_id)`.

```sql
-- Find every column derived from raw_orders.order_id
SELECT DISTINCT channel_id, output_column
FROM column_lineage
WHERE source_table = 'raw_orders' AND source_column = 'order_id';
```

## `depot.db` — Cross-run KV state

### `depot_kv`

**Source:** `aqueduct/depot/depot.py:31` — written by Egress modules with `format: depot` and read by `@aq.depot.get()` at compile time. See [specs §5.6](specs.md) and the stale-read footgun note.

| Column | Type | Notes |
|---|---|---|
| `key` | VARCHAR PRIMARY KEY | Application-defined |
| `value` | VARCHAR NOT NULL | Always serialized as string |
| `updated_at` | TIMESTAMPTZ NOT NULL | |

Note: incremental Channel watermarks are also written to `depot.db` for
backwards compatibility, but the **authoritative** location is the sidecar
file `.aqueduct/watermarks/{blueprint_id}__{channel_id}.json` (Phase 24c).

```sql
-- Inspect every depot key written in the last day
SELECT key, value, updated_at
FROM depot_kv
WHERE updated_at >= CURRENT_TIMESTAMP - INTERVAL 1 DAY
ORDER BY updated_at DESC;
```

## CLI shortcuts

Most of the queries above have first-class CLI wrappers — prefer them over
raw SQL when scripting:

| Goal | CLI |
|---|---|
| List recent runs | `aqueduct runs --last 20` |
| Show failed runs | `aqueduct runs --failed --last 10` |
| Flow report for one run | `aqueduct report <run_id>` |
| Column lineage graph | `aqueduct lineage <blueprint.yml>` |
| Gate overrides | `aqueduct signal <signal_id>` |

The raw DuckDB files are stable enough to depend on for custom dashboards —
new columns are added with `IF NOT EXISTS` migrations, existing columns are
never re-typed (except the historical `healing_outcomes.id` INTEGER → VARCHAR
migration, which auto-runs on Surveyor startup).
